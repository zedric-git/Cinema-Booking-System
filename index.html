ğŸš§ 6. Challenges and Solutions

Challenge 1: Seat Availability Management

Problem:

Initial implementation used a single SeatManager for all movies, causing seat conflicts when the same seat number was booked for different showtimes.

Solution:

Implemented dictionary-based SeatManager with composite keys:

// Dictionary keyed by "MovieTitle|ShowTime"
Dictionary<string, SeatManager> seatManagers = [];

foreach (Movie m in movies)
{
    foreach (Schedule sched in m.Schedules)
    {
        string key = m.Movies + "|" + sched.Time;
        seatManagers[key] = new SeatManager(5, 8);
    }
}

Challenge 2: Payment Deadline Enforcement

Problem:

Needed time-bound payment system with automatic expiration. Background threads caused synchronization issues and file conflicts.

Solution:

Implemented lazy evaluation approach - check deadlines only when necessary (startup, payment attempt):

// Store deadline with booking
booking.PaymentDeadline = DateTime.Now.AddMinutes(15);

// Check on startup
foreach (Booking booking in allBookings)
{
    if (booking.PaymentDeadline < DateTime.Now)
    {
        booking.PaymentStatus = PaymentStatus.Expired;
        // Release seats
    }
}

Challenge 3: Food Inventory Management

Problem:

Complex state transitions: Order selected â†’ Payment pending â†’ Payment completed â†’ Booking cancelled (refund scenario).

Solution:

Only deduct stock when payment actually completes, not on selection:

// Record sale ONLY when transitioning to paid
if (wasPending && hasFoodItems)
{
    InventoryManager.RecordSale(booking.FoodItems);
}

// Restore stock on cancellation
if (booking.PaymentStatus == PaymentStatus.Paid)
{
    InventoryManager.RestoreSale(booking.FoodItems);
}

ğŸ’¡ Key Lessons Learned:

  • Think in Keys: When multiple instances need independent state, use dictionaries with composite keys
  • Lazy is Better: Don't over-engineer with background threads when periodic checks suffice
  • State Transitions Matter: Map all possible state changes before implementing persistence
  • Test Edge Cases: Cancellations, expirations, and partial completions reveal design flaws
